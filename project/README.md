## Описание проекта

Проект реализует **end-to-end ETL-пайплайн** для загрузки данных из разнородных источников (CSV, JSON, XLSX, XML) в PostgreSQL, очистки, проверки качества, построения DWH-модели в виде **звезды** и выполнения аналитических SQL-запросов.

Это тестовое задание, демонстрирующее навыки работы с Python (pandas, SQLAlchemy), PostgreSQL, Docker и проектированием DWH.

## Архитектура и принятые решения

### 1. Staging-слой
Данные загружаются в сыром, но очищенном виде в таблицы `staging.stg_*`. Очистка включает:
- стандартизацию названий колонок (нижний регистр, подчёркивания);
- приведение типов (даты, числа);
- удаление дубликатов по первичным ключам;
- отбрасывание записей с некорректными датами (`2025-99-99`, `13/45/2025`, `broken-date`);
- замену `error_amount` и `N/A` на `NULL`.

### 2. Data Quality
Модуль `src/quality/validator.py` выполняет проверки:
- **Внешние ключи**: заказы без клиентов, платежи без заказов, события без клиентов/товаров.
- **Бизнес-правила**: отрицательные суммы/количества, недопустимые статусы заказов.
Все ошибки логируются в таблицу `staging.dq_errors` с указанием таблицы, идентификатора записи и описанием проблемы. Ошибки не блокируют загрузку, но дают прозрачность.

### 3. DWH-модель (Star Schema)
Выбрана классическая **звезда** для упрощения аналитики:
- **Измерения**: `dim_customer`, `dim_product`, `dim_date`
- **Факты**: `fact_order`, `fact_payment`, `fact_event`

Для отсутствующих внешних ключей используются **записи-заглушки** (`customer_id = 'UNKNOWN'`, `product_id = 'UNKNOWN'`).  
В `fact_payment` внешний ключ на `order_id` был удалён для упрощения загрузки, так как часть платежей не привязана к заказам, а поддерживать строгую ссылочную целостность в данном контексте не критично.

### 4. Загрузка DWH
Реализована полная перезагрузка (truncate + insert) для всех таблиц. Это допустимо для тестового задания и упрощает код. При необходимости легко переделать на инкрементальную загрузку с использованием `last_modified` или `created_at`.

### 5. Технологический стек
- **Python 3.10+** – основной язык.
- **pandas** – для работы с данными.
- **SQLAlchemy + psycopg2** – для подключения к PostgreSQL.
- **PostgreSQL 15** – СУБД (запускается в Docker).
- **Docker Compose** – для быстрого развёртывания окружения.

## Как запустить проект

### 1. Клонируйте репозиторий
```bash
git clone <your-repo-url>
cd project
```

### 2. Установите зависимости Python
Рекомендуется использовать виртуальное окружение:
```bash
python -m venv venv
source venv/bin/activate   # Linux/Mac
# или venv\Scripts\activate (Windows)
pip install -r requirements.txt
```

### 3. Настройте переменные окружения
Создайте файл `.env` в корне проекта:
```
DB_HOST=db_host
DB_PORT=db_port
DB_NAME=db_name
DB_USER=db_user
DB_PASSWORD=db_password
```

### 4. Запустите PostgreSQL через Docker Compose
```bash
docker-compose up -d
```
При первом запуске автоматически выполнятся DDL-скрипты из папки `ddl/`, которые создадут схему `staging` и `dwh` со всеми таблицами.

### 5. Разместите исходные файлы
Поместите файлы `customers.csv`, `orders.json`, `products.xlsx`, `payments.csv`, `events.xml` в папку `data/raw/` (убедитесь, что имена файлов совпадают).

### 6. Запустите ETL-пайплайн
```bash
python -m src.main
```
Скрипт выполнит:
- чтение и очистку данных;
- загрузку в staging;
- проверку качества данных (ошибки сохраняются в `staging.dq_errors`);
- загрузку DWH-таблиц.

В консоли вы увидите логи каждого этапа.

### 7. Проверьте загрузку (опционально)
Подключитесь к БД (через DBeaver или `psql`) и выполните:
```sql
SELECT COUNT(*) FROM staging.stg_customers;
SELECT COUNT(*) FROM dwh.fact_order;
```
Все таблицы должны содержать ненулевое количество записей.

### 8. Запустите аналитические запросы
```bash
python src/run_analytics.py
```
Результаты пяти запросов сохранятся в папку `reports/` в виде CSV-файлов (по умолчанию `query_1.csv` … `query_5.csv`).  
При желании можно задать свои имена файлов, добавив перед запросом комментарий `-- filename: my_report.csv` в файле `sql/analytics.sql`.

## Как воспроизвести результат

Чтобы полностью пересобрать всё с чистого листа:

```bash
# Не забыть про питон
pip install -r requirements.txt

# Остановить и удалить контейнер с томом данных
docker-compose down -v

# Перезапустить PostgreSQL с чистой БД
docker-compose up -d

# (опционально) проверить, что таблицы создались
# Затем запустить ETL
python -m src.main

# Запустить аналитику
python src/run_analytics.py
```

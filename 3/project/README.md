## Описание проекта

Проект реализует end-to-end ETL-пайплайн для загрузки данных из различных источников в PostgreSQL, очистки, проверки качества, построения DWH-модели и выполнения аналитических SQL-запросов

Всё в соответсвии с исходным заданием

Дополнительно добавлены:
- добавлена инкрементальная загрузка DWH-таблиц
- Airflow для оркестрации пайплайна по расписанию
- unit-тесты для функций трансформации

## Как запустить

### Нужно:

### 1. Настроить переменные окружения `.env`
```bash
DB_HOST=postgres

# или/или 
# в зависимости от окружения 

DB_HOST=localhost
```
Пример в `.env.example`

### 2. Установить зависимости
```bash
python -m venv venv
source venv/bin/activate 
pip install -r requirements.txt
```

### 3. Запустить PostgreSQL и Airflow через Docker Compose
```bash
docker-compose up -d
```
При первом запуске автоматически
- создадутся БД и таблицы
- инициализируется Airflow
- поднимутся веб-сервер и планировщик

### 4. Открыть Airflow
- `http://localhost:8080`  
- В списке DAG-ов появится `etl_dag`

### 5. Запуск ETL вручную
Если нужно выполнить загрузку один раз:
```bash
python -m src.main
```

### 6. Выполнение SQL-запросов
```bash
python -m src.run_analytics
```
- Результаты пяти запросов сохранятся в папку `reports/` в виде CSV-файлов 


### 8. Запуск unit-тестов
```bash
pytest tests/
```

## Архитектурные решения

### 1. Staging-слой
Данные загружаются в очищенном виде в таблицы `staging.stg_*`

Очистка включает:
- стандартизацию названий колонок
- приведение типов
- удаление дубликатов по первичным ключам
- отбрасывание записей с некорректными датами
- замену `error_amount` и `N/A` на `NULL`
- приведение идентификаторов к строке без десятичной точки

### 2. Data Quality
Модуль `src/quality/validator.py` выполняет проверки:
- Внешние ключи:
  - заказы без клиентов, платежи без заказов, события без клиентов/товаров
- Бизнес-правила: 
  - отрицательные суммы/количества, недопустимые статусы заказов

Все ошибки логируются в таблицу `staging.dq_errors` с указанием таблицы, идентификатора записи и описанием проблемы

> Ошибки не блокируют загрузку

### 3. DWH-модель
*Star Schema по заданию*

**Измерения**:
- `dim_customer`
- `dim_product`
- `dim_date`
  
**Факты**: 
- `fact_order`
- `fact_payment`
- `fact_event`

Реализована инкрементальная загрузка

- таблица `dwh.etl_control` хранит время последней успешной загрузки для каждого факта и измерения

При каждом запуске загружаются только новые записи

Для отсутствующих внешних ключей используются записи-заглушки `UNKNOWN`

В `fact_payment` внешний ключ на `order_id` был удалён для упрощения загрузки, т.к. часть платежей не привязана к заказам

### 4. Airflow
Пайплайн упакован в DAG `etl_dag` - запуск ежедневно

Airflow использует ту же БД для хранения метаданных

Веб-интерфейс на `http://localhost:8080`

### 5. Unit-тесты
Для функций очистки `transform.py` написаны тесты с использованием `pytest`

Проверка
- стандартизации колонок
- удаления дубликатов
- обработки некорректных дат и чисел
- замены пустых строк на `pd.NA`

## Как воспроизвести результат

```bash
# Не забыть про .env
# Установить зависимости
python -m venv venv
source venv/bin/activate 
pip install -r requirements.txt

# Остановить и удалить всё
docker-compose down -v

# Запустить заново
docker-compose up -d

# Запустить ETL вручную или через Airflow
python -m src.main

# Или активировать DAG в веб-интерфейсе Airflow

# Выполнить аналитику
python -m src.run_analytics
```
